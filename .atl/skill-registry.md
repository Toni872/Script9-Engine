## Compact Rules

Pre-digested rules per skill. The orchestrator copies matching blocks into sub-agent prompts as `## Project Standards (auto-resolved)`.

### work-unit-commits

- Plan implementation as a sequence of reviewable work units BEFORE writing any code.
- One commit per logical change; tests and docs ship WITH the code that needs them — never in a separate "cleanup" commit.
- Use Conventional Commits: `feat:`, `fix:`, `chore:`, `refactor:`, `test:`, `docs:`, `build:`, `ci:`. Imperative mood, subject ≤ 72 chars.
- NEVER add `Co-Authored-By` or AI attribution. No `--no-verify`, no `--amend`, no `git config` mutations.
- If a change crosses 400 lines or touches 3+ subsystems, split into a chained PR stack.
- Commit body explains WHY (the rationale and the gotcha), not WHAT (the diff already shows that).

### branch-pr

- Run `issue-first` checks: open/verify a GitHub issue BEFORE writing code; reference it in the PR.
- Branch name format: `feat/<scope>-<short-desc>`, `fix/<scope>-<short-desc>`, `chore/<scope>-<short-desc>`. Lowercase, hyphens, no author handles.
- PR title matches the lead commit's subject; PR body has: summary, linked issue (`Closes #N`), test plan, screenshots if UI.
- Keep PRs under 400 changed lines when possible; over 400 → chained PR via `chained-pr` skill.
- Use `gh` CLI for everything GitHub. Return the PR URL when done.

### chained-pr

- Trigger: PR > 400 lines, stacked PRs, or 3+ subsystems touched in one branch.
- Split into a stack of dependent branches: `1-base`, `2-feature-a`, `3-feature-b`, etc. Each must build and test green independently.
- Each chained PR has its own review thread; main branch is the LAST in the chain, intermediate branches target the previous chain link, not main.
- Rebase-friendly: keep commits linear (no merge commits inside the stack).
- Communicate the chain order clearly in each PR description ("Depends on #N").

### issue-creation

- Issue-first: open an issue BEFORE writing code that closes it. The issue is the contract.
- Title format: `<area>: <imperative summary>` (e.g. `billing: handle Stripe portal return for cancelled subscriptions`).
- Body sections: Context, Problem, Proposed Solution, Acceptance Criteria, Out of Scope, Related Links.
- Acceptance criteria are testable. Each one maps to a scenario in the spec or a test case in the apply phase.
- Use labels: `area:*`, `priority:*`, `type:*` consistently.

### comment-writer

- Tone: warm, direct, caring — like a senior teammate who actually wants you to grow.
- Lead with the WHAT and WHY, then the suggestion. Don't bury the actionable in prose.
- Distinguish blocking (must fix) from non-blocking (nit, suggestion, question). Use prefixes: `nit:`, `q:`, `blocking:`, `suggestion:`.
- For "this is wrong" cases: (1) acknowledge the question makes sense, (2) explain WHY with evidence, (3) show the correct way with an example.
- No sycophancy, no empty praise, no "great catch!" filler. Just substance.

### cognitive-doc-design

- Lead with the user's goal and the smallest usable example. Build up from there.
- One concept per section. Use headers as a table of contents the reader can scan in 5 seconds.
- Show the failure mode before showing the fix — readers remember problems better than solutions.
- Code examples must compile/run as written. Include imports, file paths, and a one-line "before / after" delta.
- Reference the actual codebase (paths, line numbers) instead of abstract descriptions.
- For long docs (like the project's 822-line `Script9_Engine.md`): front-load the table of contents and the 30-second summary; the rest is deep reference.

### judgment-day

- Run two independent reviewers in fresh context. Do NOT do the review yourself in the main thread.
- Reviewer A: code-quality / security / correctness. Reviewer B: spec-conformance / UX / consistency.
- Collect verdicts, surface only CONFIRMED issues (both reviewers flagged, OR single reviewer flagged with high severity).
- Fix confirmed issues, then re-run a second-round review (cheap, catches regressions).
- Output format: table of issues with severity, location, evidence, and fix.

### skill-creator

- LLM-first skill: frontmatter is the contract — `name`, `description` (≤ 250 chars, trigger-first, one physical line), `license`, `metadata`.
- Required structure: Activation Contract, Hard Rules, Decision Gates, Execution Steps, Output Contract, References.
- Hard rules must be observable (verifiable in output). Decision gates cover real forks. Output contract states exactly what to return.
- Keep body 180–450 tokens. Move examples, schemas, edge cases to local `references/` or `assets/`.
- Compact rules (5–15 lines) for sub-agents: actionable constraints, key patterns, breaking changes, gotchas. NO purpose, motivation, full examples.
